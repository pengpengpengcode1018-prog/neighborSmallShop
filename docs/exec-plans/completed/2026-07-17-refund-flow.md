# 退款申请、审核与微信退款

- 状态：Completed
- 日期：2026-07-17
- 关联功能：`refund-flow`
- 前置计划：`../completed/2026-07-17-expired-order-close.md`

## 目标

居民可对规则允许的已支付订单提交退款申请，管理员可审核并安全发起微信退款；累计退款不超过实付金额，重复请求、通知和外部未知结果都可恢复且可审计。

## 范围

- 可视复核需求说明书退款、订单状态、后台审核和安全验收页，并复核微信支付官方退款申请、退款查询和退款结果通知契约。
- 定义第一版 `PAID -> REFUND_PENDING -> REFUNDED` 状态边界、申请资格、审核权限、拒绝语义和履约竞态；已接单售后明确延期。
- 增加退款单、唯一请求号、累计金额、微信退款号、状态、失败原因、通知摘要和操作审计 migration。
- 以 provider/repository/service 分层实现申请、审核、微信退款、主动查询和通知恢复；金额完全由服务端 Decimal 计算。
- 交付居民小程序申请/进度与管理后台审核入口，并覆盖重复、并发、超额、越权和网络未知状态。

## 非目标

- 不执行真实生产退款，不提交商户私钥、APIv3 key 或真实用户数据；真实商户与真机验证仍是发布门禁。
- 不实现部分商品售后、多次部分退款、人工转账、退货物流或客服工单；第一版只允许一单一次、按服务端实付金额整单退款。
- 不交付订阅消息、后台实时提醒、经营看板或生产任务调度平台。

## 风险与开放决策

- 退款金额、订单状态与微信退款结果跨系统一致，不能在外部结果未知时伪装成功或重复累计。
- 审核通过与继续履约可能并发；必须以明确状态条件和行锁确定唯一有效动作。
- 原需求只给出居民“取消已支付未接单订单”和管理员审核入口，没有部分金额输入；第一版据此固定为一单一次整单全额退款，数据库仍校验累计退款不超过实付金额。
- 退款通知与主动查询需要共用同一幂等终态事务，且不保存通知原文、完整手机号或支付密钥。

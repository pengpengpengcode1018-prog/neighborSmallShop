# 经营看板与操作日志

- 状态：Completed
- 日期：2026-07-17
- 关联功能：`operations-dashboard-audit`
- 前置计划：`2026-07-17-notifications-alerts.md`

## 目标

为平台管理员提供口径明确、可验证的经营看板和去敏、不可删除的操作记录，使订单、成交、用户、店铺与商品指标能够追溯到服务端真实数据。

## 已完成范围

- 可视复核需求说明书第 4-5、22、30-31、38、47-48 页，并冻结上海业务日、有效支付订单及支付/退款/净额口径。
- 实现 `/api/v1/admin/operations/dashboard` 服务端聚合，提供近 7/30 天趋势、今日履约状态和店铺表现，金额按 Decimal 两位小数返回。
- 实现后台 ECharts 经营看板、口径说明、店铺表现，以及加载、空、错误和重试状态。
- 后台写审计统一经过字段白名单和文本去敏，新增独立 request id；小区详细地址、手机号、Token/密钥、备注与自由文本详情不进入 diff。
- 实现操作日志筛选、分页和详情 API/后台页面；列表不返回 diff，详情再次去敏，且生产路由不提供修改或删除。
- 为支付成功、退款完成和 request id 增加查询索引，同步产品规格、可靠性、安全、架构与数据库文档。

## 明确非目标

- 未实现财务对账、商家结算、导出报表、BI 平台、实时流处理或任意 SQL 查询。
- 未伪造此前不存在的操作日志，也未接入缓存或汇总表作为新的统计真相。
- 未向后台列表暴露居民手机号、地址、微信身份或支付/退款原始通知。

## 关键决策

- 今日订单按创建时间统计并包含未支付/取消；有效支付订单按支付时间统计且排除退款成功，退款审核中仍视为有效。
- 净成交额按业务日内“成功支付金额 - 退款成功金额”计算，并同时返回支付、退款和净额，避免退款影响被隐藏。
- 趋势只开放 7/30 天有界窗口；第一版直接读取交易真相，数据量证明需要后才引入汇总或缓存。
- before/after 采用显式业务白名单而非黑名单兜底；写入前和详情返回前各去敏一次，列表不加载 JSON。
- OperationLog 是只读审计证据；登录成功/失败仍由专用 `AdminLoginLog` 保存，业务操作使用 `OperationLog`。

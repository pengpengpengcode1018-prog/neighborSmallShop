# 配送时间与容量管理

- 状态：Completed

## 目标

管理员可按店铺配置尽快送达、预约配送和每个预约送达时刻的订单上限，为居民下单时的可用时段查询和容量占用建立可信数据边界。

## 完成范围

- 使用 PDF 页面截图核验需求说明书第 14、15、26、27、47、49、52 页，修正每日固定送达时刻的产品语义。
- 增加店铺“尽快送达/预约配送”开关和 `DeliverySlot` 模型，应用 `20260716084500_delivery_slots` migration。
- 实现配置读取、配送方式更新、时段创建/编辑/启停 API，以及事务内操作审计。
- 覆盖未授权、至少一种方式、截止时间、容量下限、重复时刻、跨店访问、编辑、停用和审计测试。
- 实现后台店铺切换、配送方式、时段列表、表单、启停、空态、错误重试和可访问开关名称。

## 关键决策

- “尽快送达”是独立配送方式，不作为特殊预约时段存储；店铺至少启用一种配送方式。
- 预约配置是每日重复的固定送达时刻，没有星期或起止区间；停止下单时刻必须早于同日送达时刻。
- 同一店铺的送达时刻由数据库唯一索引保证不重复，最大订单数使用无符号小整数且必须大于零。
- 原始需求只定义启停，不定义删除；第一版不暴露时段删除 API，停用后保留配置和未来历史引用边界。
- 当前切片只配置容量，不提前实现订单占用；后续下单事务仍以数据库为最终并发边界。

## 遗留风险

- 按星期、节假日例外和跨午夜时段没有产品行为，需在明确需求后扩展，不能静默解释为当前每日时段。
- 订单容量占用、取消释放和并发上限仍未实现，必须随订单 migration 和事务测试一起进入。
- 本计划未调用真实微信能力，也未使用生产数据或凭证。

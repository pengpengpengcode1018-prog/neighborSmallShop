# 可靠性

## 标准路径

- Bootstrap：`./init.sh`
- 全量验证：`npm run verify`
- 基础设施：`npm run infra:up`，检查 `docker compose ps`
- API：`npm run dev:server`，先访问 `GET http://localhost:3000/api/v1/health`，再访问 `GET http://localhost:3000/api/v1/ready`
- 后台：`npm run dev:admin`
- 小程序：`npm run dev:miniapp`，用微信开发者工具打开 `miniapp/dist/dev/mp-weixin`
- 发布静态门禁：`npm run verify:release`
- API 生产锁在线审计：`npm run release:audit:runtime`；仓库构建链审计：`npm run release:audit`
- 隔离发布演练：`npm run release:rehearse`；临时随机凭证、容器、网络和数据卷默认在结束时清理
- 阿里云生产部署：按 `docs/PRODUCTION_DEPLOYMENT.md` 使用外部 RDS、ECS 内部 Redis 和 `compose.production.yaml`，先候选端口验证再切宝塔反代

## 必需信号

- API 请求具有 request id、结构化完成日志和统一错误响应。
- 外部 request id 只接受 1-128 位字母、数字、点、下划线、冒号和连字符，其他值由服务端 UUID 替换；请求体只接受 JSON、最大 1 MiB，微信通知另受 128 KiB 上限约束。
- 普通接口与身份入口有独立固定窗口频率上限；超限返回 HTTP 429 与 `RATE_LIMITED`、`Retry-After`，单进程桶最多保留 10,000 个来源。
- `/api/v1/health` 不访问外部依赖，只证明 API 进程存活。
- `/api/v1/ready` 在 2 秒探测预算内分别验证 MySQL 查询和 Redis PING；任一依赖不可用时返回 HTTP 503 与 `SERVICE_UNAVAILABLE`。
- 支付回调、退款、定时关单、库存恢复和后台操作保留去敏审计日志。
- UI 明确提供 loading、empty、success、error 和 retry 状态。
- 微信登录 provider 使用 5 秒超时并将无效 code、限流、缺少凭证、网络及格式故障归一为稳定错误；外部原始响应不进入日志或客户端。
- 微信手机号 provider 复用带提前失效窗口的 access token 缓存，以 5 秒超时调用官方动态 code 接口；token 失效只刷新重试一次，无效 code、限流和平台不可用分别归一为稳定 400、429 和 503。绑定同一用户可安全重试，号码已属于其他账号时返回 409 且不泄露归属。
- 小区公共列表是有效性的校验源；网络失败保留本地原选择，服务端确认停用或删除时清除失效选择并要求用户重选。
- 居民店铺列表在每次小区切换后重新查询服务端；请求竞态只接纳最新小区响应，失败时不保留可能属于旧小区的店铺结果。
- 居民分类切换只接纳最新分类响应；商品展示状态和 `canPurchase` 由服务端结合库存、上下架、店铺营业与配送上下文给出，客户端不从缓存状态重建资格。
- 购物车以 MySQL 服务端记录为真相；每次写入重验账号当前小区、店铺营业/配送、商品状态、库存和限购，客户端仅在服务端确认成功后替换本地状态，网络失败保留上一次已确认结果。
- 跨店加购默认只返回冲突，必须由用户确认后携带显式替换字段在单个事务中清空并加入；读取时保留后来失效的项目和原因，不静默丢失用户选择。
- 地址列表以 MySQL 为真相且只存在当前居民内存中，不持久化到小程序本地存储；退出登录立即清除姓名、手机号和详细地址。地址写入重验平台小区，失效地址保留并标记恢复动作。
- 新建地址在恢复本人资料后只对空白联系电话预填账号绑定手机号；编辑已有地址和用户手工修改后的值不被覆盖。未绑定时由原生授权按钮显式重试，拒绝不会生成占位号码或地址。
- 默认地址变更先锁定居民记录，并由可空唯一 `default_key` 约束兜底；删除默认地址时按最近使用、最近编辑顺序确定性选出替代地址。
- 订单预览和创建都从服务端购物车重新校验商品、金额、地址、店铺与配送；客户端回传预览版本和金额，变化时必须重新确认，不静默使用旧价格。
- 订单创建先按居民加行锁并检查唯一请求 id，相同参数重试返回原订单；网络失败时小程序保留完全相同的创建请求用于安全重试。
- 预约创建在任何一致性读之前锁定店铺时段行，再按日期统计未取消订单；库存用带下限条件的更新，容量、库存、快照、状态日志、地址最近使用和购物车清理处于同一事务。
- 居民和管理员订单列表、详情都读取订单快照和同一状态真相；居民查询按当前用户隔离，后台列表手机号默认脱敏，客户端失败时保留上一次服务端确认结果并允许重试。
- 待付款取消/关闭先用短事务领取支付关闭租约，离开事务执行微信查单/关单，再锁定订单提交状态、`stock_released`、库存恢复和审计；重复或竞态请求不重复恢复库存或写日志。
- 微信支付初始化以订单金额、订单号、OpenID 和到期时间为服务端真相；同一订单只创建一条支付记录，并发初始化返回明确处理中或复用仍有效的 `prepay_id`。
- 微信通知在 128 KiB 上限内先使用原始请求体、时间窗口和 `Wechatpay-Serial` 对应的平台公钥验签，再以 APIv3 key 解密；轮换窗口可同时接受已配置的旧、新 ID，但未知 ID 或 ID/签名错配必须失败。商户/AppID/订单/币种/金额全部匹配后，支付、订单、销量、通知摘要和状态日志在同一事务提交。
- 重复通知和可信主动查单共用同一入账事务，不重复增加销量；小程序支付面板的成功、取消或失败都必须再读服务端状态，无法确认时显示未知而不猜测成功。
- 居民取消、后台关闭和 15 分钟超时任务共用安全关单服务：`SUCCESS` 进入统一入账事务，`NOTPAY` 关单，`CLOSED` 提交本地关闭，其他或网络未知保留库存等待重试。
- 超时任务启动后立即执行并按配置间隔、批量上限扫描；进程内阻止批次重叠，支付关闭租约阻止多 worker 重复外部调用，租约过期后可由重启进程继续恢复。
- 后台履约动作使用期望状态条件更新，并在同一事务中写订单状态日志和管理员操作日志；生产 API 不提供手工写入 `PAID` 的入口。
- 居民退款申请只接受本人 `PAID` 未接单订单，请求号和服务端参数摘要保证网络失败安全重放；申请事务锁定订单并与后台接单竞争，只有一个动作成功。
- 退款审核先持久化固定商户退款单号，再离开事务调用微信；网络未知时保留 `APPROVED|PROCESSING` 和库存，通过同一退款单号查询或重试恢复，不能猜测成功。
- 微信退款申请响应、主动查单和验签解密后的通知共用终态事务；交易号、订单号、退款号、币种和金额全部匹配后，`SUCCESS` 才恢复一次库存、扣回一次销量并结束订单。并发审核使用 `READ COMMITTED` 与订单/退款行锁避免等待锁后读取旧快照。
- 订阅消息 worker 从订单状态日志幂等发现 outbox；发送前原子领取事件并预占一次客户端报告授权。微信明确未发送的可恢复错误才归还授权并延迟重试，消息请求网络结果未知时进入 `UNKNOWN` 且不自动重试。
- 进程中断遗留超过 5 分钟的 `SENDING` 事件标记为 `UNKNOWN`，不会猜测未发送；模板未配置、居民禁用或无报告授权记录为稳定跳过原因。消息与提醒都不参与订单状态事务的真相判断。
- 后台提醒任务和查询共用幂等刷新：新单取支付状态日志，超时取仍为 `PAID` 且超过 10 分钟的订单，退款取待审核记录，库存取服务端预警阈值；业务状态恢复时自动解决，重新触发时可恢复未读。
- 经营看板以服务端上海业务日窗口读取成功支付、成功退款和当前履约状态；趋势窗口只允许 7/30 天，金额始终用 Decimal 累计并返回两位小数，客户端不从分页列表重算。
- 后台写审计统一经过字段白名单和文本去敏后落库；列表不加载 diff，详情再次去敏，request id/IP/路径用于跨日志追踪。生产路由不提供操作日志更新或删除能力。
- 后台居民用户列表直接读取居民权威记录；首次微信身份登录后立即可见，手机号绑定更新同一行。列表支持状态、绑定状态、小区和关键词筛选，列表与详情都只返回脱敏手机号及最小资料。

## 黄金旅程

1. 居民微信登录 -> 本服务创建/恢复会话 -> 用户点击授权手机号 -> 新建地址自动预填；拒绝后保留浏览会话并可再次授权。
2. 管理员登录 -> 建小区 -> 建店铺 -> 配配送范围 -> 建分类和商品。
3. 用户选小区 -> 浏览单店 -> 加购物车 -> 选地址与时段 -> 创建待支付订单。
4. 微信支付初始化 -> 回调或查单确认 -> 后台接单和配送状态流转 -> 用户看到完成。
5. 可取消或退款的订单 -> 幂等处理 -> 库存和金额一致。
6. 居民主动授权订单通知 -> 状态事件异步投递；管理员开启本次会话提醒音 -> 新单/超时提醒可见且业务状态不被提醒修改。
7. 管理员打开居民用户列表 -> 查看首次登录记录和手机号绑定状态；再打开经营看板核对支付/退款口径，并按 request id 或业务 ID 查询只读操作日志。

每条旅程在进入实现计划时必须补 API 集成测试；用户可见流程还需浏览器或微信开发者工具证据。

## 故障与重启

- 进程重启不丢失业务真相；MySQL 为权威数据源，Redis 只承载可重建状态或有明确恢复策略。
- 同一外部回调和同一请求 id 可安全重放。
- 某类故障重复两次以上时，将其升级成测试、健康信号或 guardrail。
- `npm run verify`、容器重启和服务冷启动未通过时，功能不能标记 passing；冷启动必须使用 `npm run build` 后的产物并同时验证 `/health` 与 `/ready`。
- `compose.release.yaml` 同时提供两个仅绑定 `127.0.0.1` 的入口：API 默认 `3100 -> server:3000`，后台默认 `8080 -> admin-web:8080`；宿主 Nginx 分别以 `api.surroundsmallshops.com` 和 `www.surroundsmallshops.com` 终止 TLS，MySQL/Redis 仍只在内部网络。
- 发布演练为 API 和后台分配独立随机 loopback 端口，从空卷等待 MySQL/Redis、运行 14 个迁移，验证 API readiness、后台同源代理、401、运行镜像不含 Prisma CLI，再重启 API 并复验 readiness。
- Docker 构建使用宿主构建网络和带锁 BuildKit npm 缓存以应对下载抖动；该权限仅存在于镜像构建阶段，不改变运行容器的内部网络隔离。
- 公网验收不能只看 HTTP 状态：`api` 的 `/api/v1/health` 必须返回 `code=0` 且 `data.service=nearby-shop-server`，`www` 首页标题必须是“近邻小铺子管理后台”；否则视为反代仍指向旧应用。

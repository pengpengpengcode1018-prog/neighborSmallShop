# 范围、角色与默认决策

来源：需求说明书第 1-5、47-52 页。

## 用户与角色

- 居民通过微信小程序登录、选择小区、购物、支付、管理地址和查看配送状态。
- 平台管理员通过 Web 后台维护全部店铺、商品、配送范围/时间、用户、订单、退款和经营数据。
- 店铺管理员是第二阶段角色；第一版只预留 RBAC 与店铺范围，不提供独立后台或权限入口。

## 第一版闭环

选小区 -> 浏览店铺和商品 -> 单店购物车 -> 收货地址和配送时间 -> 创建订单 -> 微信支付 -> 后台接单 -> 准备和配送 -> 完成。

## 必须保持的默认决策

- 平台统一创建店铺和商品；商家自行备货配送。
- 多店铺展示，但购物车、订单和支付均是单店铺。
- 用户手选固定小区；不使用地图自动定位或实时轨迹。
- 单规格商品；店铺独立配置范围、起送额、配送费与时段容量。
- 待支付订单 15 分钟关闭，支付成功后才计入有效订单和成交金额。

## 身份与会话

- 居民以微信 `uni.login` 临时 code 登录；服务端经微信登录凭证校验取得身份后创建或恢复本服务 JWT 会话。
- 首次微信登录由“微信身份会话”和“手机号授权”两个连续步骤组成：身份会话建立后立即引导用户点击微信 `getPhoneNumber` 授权按钮，服务端使用一次性动态 code 调用微信接口取得并绑定经过平台校验的手机号；不得静默读取、要求用户手工抄录微信手机号或信任客户端直接提交的号码。
- 用户拒绝、取消或授权失败时不写入占位手机号，保留微信身份会话并允许继续浏览，但购物车、地址、订单、支付和退款等个人或交易能力继续提示完成手机号授权；重试成功后无需重新微信登录。
- 微信身份建立后，居民可以主动选择微信头像并填写昵称完善可展示资料；该步骤可跳过，拒绝或未填写不得阻断浏览、购物和下单。服务端保存用户确认后的昵称与压缩头像，首页登录区和后台居民列表/详情展示同一份资料，未完善时回退为“微信用户”和文字头像。
- 头像只接受居民主动选择的 PNG、JPEG 或 WebP 图片，单张不超过 512 KiB；服务端保存受控副本并通过居民头像接口返回，不持久化微信临时文件地址，也不得静默读取头像或昵称。
- 当前居民资料接口只向本人返回昵称、头像地址、已绑定的完整手机号和 `phoneBound` 状态，用于登录后展示与地址表单预填；后台只返回用户主动保存的昵称、头像地址和脱敏手机号，任何日志和错误响应默认不返回手机号、头像内容或微信身份标识。
- 游客可浏览店铺和商品；个人资料、服务端购物车、地址、订单、支付及退款等个人或交易能力必须使用有效用户会话。
- 每次受保护请求都重新确认用户仍为启用状态；管理员 token 与居民 token 不可互用。
- 微信登录 code、手机号动态 code、AppSecret、session key、OpenID、JWT 和完整手机号不得进入日志、错误响应、截图或文档。

## 明确不做

商家入驻/独立后台、配送员端、复杂 SKU、优惠券和营销、评价积分、结算抽佣、多店支付、发票和完整售后工单系统。

## 验收

- 每个第一版功能都能追溯到功能清单、规格和验证证据。
- 非目标不以隐藏入口、半成品表或默认开启的方式进入用户界面。
- 新范围先更新规格和计划，再实现。
